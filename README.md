# CS250
**This repository contains my Sprint Review and Retrospective deliverable for CS 250. The artifact reflects on the Scrum-Agile practices used throughout the SNHU Travel project, including Scrum roles, user stories, backlog refinement, testing, communication, and responding to changing requirements.**
## Essential Questions Reflection
- How do I interpret user needs and implement them into a program? How does creating user stories help with this?
  - I interpret user needs by looking for the goal behind the request, not just the surface-level feature being asked for. A user may describe what they want in broad or unclear terms, so it is important to identify who the user is, what they need to accomplish, and why that need matters. In the SNHU Travel project, user feedback and stakeholder input had to be translated into clearer development tasks before the team could build or test anything effectively.
  - Creating user stories helps with this process because user stories connect development work to user value. A user story describes a feature from the user’s perspective, usually by identifying the user, the desired action, and the benefit. This helps the team avoid building features only because they are technically possible. Instead, the team can focus on whether the work supports the user’s goal. User stories also make it easier to define acceptance criteria and test cases, which helps the team know when a feature is complete.
- How do I approach developing programs? What Agile processes do I hope to incorporate into my future development work?
  - I approach developing programs by breaking large problems into smaller, manageable pieces. Instead of trying to solve an entire project at once, I try to identify the most important requirements, clarify what the program needs to do, and work incrementally. This approach helps reduce confusion and makes it easier to test progress as the program develops.
  - In future development work, I hope to incorporate Agile practices such as user stories, backlog prioritization, iterative development, regular testing, and retrospectives. User stories and backlogs help organize work around user value. Iterative development makes it possible to create a working version of the product, receive feedback, and improve it over time. Testing throughout the process helps catch problems earlier. Retrospectives are also useful because they encourage the team to reflect on what worked, what did not work, and what should change in the next cycle.
- What does it mean to be a good team member in software development?
  -  a good team member in software development means communicating clearly, taking responsibility for assigned work, and supporting the success of the team rather than only focusing on individual tasks. Software development depends on collaboration because requirements, design, coding, testing, and feedback are connected. If one team member does not communicate progress or blockers, the rest of the team may make decisions based on incomplete information.
  - A good team member should ask clarifying questions, share updates, document important decisions, and be willing to adapt when requirements change. In a Scrum-Agile environment, this also means respecting team roles and Scrum events. The Product Owner, Scrum Master, developers, and testers all contribute to the final product in different ways. A strong team member understands how their work affects others and helps create transparency so the team can deliver a useful product increment.
